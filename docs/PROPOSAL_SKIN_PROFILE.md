# 제안: 피부타입 프로필 + 얼굴 분석 연계 (2026-07-22)

얼굴인식(AI 피부 진단)을 단독 기능이 아니라 **"피부타입 프로필을 채우는 입력 수단"** 으로
재정의하고, 프로필이 있으면 추천에 자동 반영하자는 제안입니다.
근거는 [`RELATED_WORK.md`](RELATED_WORK.md)의 조사 결과를 기반으로 합니다.

## 제안 요지

> **프로필이 본체, 얼굴인식은 입구.**

- 사용자가 피부타입(지성/건성/복합성/민감성)과 고민을 프로필로 저장하면 채팅 추천에 자동 반영
- 얼굴 분석은 그 프로필을 쉽게 만들어주는 **선택적** 수단 — 분석 결과는 "확정"이 아니라
  **"피부타입 제안"** 으로 주고, 사용자가 확인·수정 후 저장
- 현재 `app/face-analysis/`(개발 중 플레이스홀더)가 추천 흐름과 단절된 단독 기능인 문제를 해소

### 2. 시장도 같은 방향

- **잼페이스** — AI 셀카 피부 분석 → 제품 적합도 판정, 진단 이용 150만회 돌파 (수요 검증)
- **화해 "궁합 점수"** — 피부타입·고민 프로필 설정 → 적합도 수치화. 업계가 이미
  "프로필 기반 적합도"로 이동 중

### 3. 프로필 우선 설계가 리스크를 낮춤

| 리스크 | 프로필 우선 설계의 해소 방식 |
|---|---|
| 얼굴 사진 = 민감정보(생체정보) | 기본은 자가선택 프로필, 얼굴 분석은 선택 + 사진 서버 미저장 |
| 이미지 분석 일관성 (교수님 피드백 2번 — 반복 편차) | 조명·카메라 편차가 큰 이미지 분석 결과를 "확정"이 아닌 "제안 → 사용자 확인"으로 설계해 회피 |
| "AI는 계산하지 않는다" 원칙 | 프로필 반영은 AI가 아니라 코드 레벨 보정(`lib/safety.ts` 확장)으로 처리 — 결정론 유지 |

## 개발 방법 (3단계)

### 1단계 — 피부타입 프로필 (핵심, 먼저)

- 지성/건성/복합성/민감성 + 고민 자가선택 → `localStorage` 저장
  - 즐겨찾기·비교함과 동일 패턴, 로그인 불필요, 기존 `lib/useLocalStorage.ts` 훅 재사용
- `/api/chat`, `/api/recommend` 요청 바디에 `skinType` 필드 추가 → 프로필 있으면 자동 반영
- 얼굴 분석 없이도 이 단계만으로 기능이 완결됨 (작업량 최소)

### 2단계 — 점수 반영 (결정론 유지)

- `lib/safety.ts` 확장: 민감성 프로필이면 자극 성분 순위 강등을 더 강하게
  (기존 안전장치의 자연스러운 확장 — AI 무관, 항상 동작)
- Gemini 분류 프롬프트(`lib/gemini/analyzer.ts`)에 프로필 컨텍스트 추가 → 분류 정확도 향상
  (점수 계산은 여전히 `lib/scoring/calculator.ts` 순수 산식)

### 3단계 — 얼굴 분석 = 프로필 입력 수단으로 연결

- `app/face-analysis/` 페이지를 "프로필 만들기" 흐름으로 연결
- 분석 결과는 **피부타입 제안** → 사용자가 확인·수정 후 저장
- 사진은 서버 전송 없이 처리 (페이지에 이미 "서버에 저장 안 함" 안내 존재), 의료 진단 아님 면책 유지

## 검증 계획

- 1·2단계: 동일 고민 입력에 대해 프로필 유/무·타입별 추천 결과 차이를 스냅샷 비교
  (산식이 결정론이므로 단위 테스트 가능)
- 3단계: `scripts/consistency-check.mjs` 방식을 확장해 동일 이미지 반복 분석 편차 측정
  — 편차가 크더라도 "제안 → 확인" 설계라 통과 기준을 완화해 적용 가능
